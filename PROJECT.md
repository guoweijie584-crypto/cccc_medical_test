## 2. 项目定位与边界

### 一句话定位

> 面向血糖管理咨询场景的多智能体协作系统，通过记忆治理与人类评价驱动受控优化。

### 项目性质

- 课程/研究原型项目（九安医疗课题）
- 目标场景：智能客服环境下的慢病（糖尿病）血糖管理咨询
- GitHub 仓库：`github.com/guoweijie584-crypto/cccc_medical_test`

### 三个核心能力

| # | 能力 | 说明 |
|---|------|------|
| 1 | **多智能体协作服务** | 不同专业角色协作给出综合建议，避免单模型泛化回答 |
| 2 | **记忆治理** | 让系统能正确读取、组织、更新患者长期上下文，而不是做"三层记忆"概念展示 |
| 3 | **评价驱动的受控优化** | 在人工评价和规则约束下持续改进，而不是无约束自进化 |

### 明确"不再"的方向

以下方向在早期探索中已明确放弃或降级，**不得在新代码/文档中重新引入**：

- ~~不再把三层记忆当主线~~ —— 记忆系统是"统一底座 + 治理"，不是"短期/中期/长期"三个独立模块
- ~~不再讲完全自进化~~ —— 是"人工评价约束下的受控优化"，不是系统自主无限自改
- ~~不再用万能 Memory Agent 管一切~~ —— Memory Governor 只管记忆读写与治理，不做路由决策或替代其他 Agent 思考

### 当前阶段不追求

- 完整生产级部署（我们是研究原型）
- 全自动 prompt 自改写（先做人工评价驱动）
- 真正临床诊断（我们给出"咨询建议"，不做"诊断结论"）
- 不可验证的研究机制堆砌（不做无法验证效果的功能）

## 3. 总体设计原则

### 3.1 两条环路必须分开

整个系统由两条环路组成，**设计、代码、接口、任务分工都按两条环独立组织**：

#### 在线服务环（实时 — 用户真正看到的链路）

```
用户输入 → Coordinator 重述/分类 → Memory Governor 读取上下文
→ 动态调度 Specialist Agent(s) → 专家并行输出结构化意见
→ Safety Reviewer 安全检查 → Coordinator 汇总最终答复 → 返回用户
→ Memory Governor 写回值得保留的信息 → 生成待评价记录
```

#### 离线优化环（非实时 — 系统持续改进的链路）

```
收集咨询日志/专家意见/记忆调用记录/最终答复
→ 人类专家评价与标注（GOOD / BAD / NEUTRAL / ERROR）
→ 分析失败类型与原因
→ 形成候选优化方案（仅允许优化的对象）
→ 离线测试集/回放集验证
→ 通过后进入下一版
```

### 3.2 统一记忆底座 + 记忆治理

- 存储层：Memory Palace（SQLite + 混合检索），提供统一存取接口
- 治理层：Memory Governor Agent 负责读取、组织、写回、冲突处理、新鲜度管理
- **不是三层记忆**：不按"短期/中期/长期"划分独立模块，而是通过治理策略管理不同生命周期的信息

### 3.3 评价驱动优化

- **允许被优化的对象**：Prompt 模板、路由规则、记忆抽取模板、上下文组织策略、专家调用策略
- **不允许系统自动改的对象**：安全红线、医疗高风险规则、人工评价标准、关键阈值与升级机制
- 优化流程：人工标注 → 失败分析 → 候选修改 → 离线验证 → 通过后才入版

### 3.4 设计围绕"能力边界"而不是"仓库结构"

- 以"系统能做什么/不能做什么"为核心来组织设计，而不是以"代码放在哪个文件夹"来驱动开发
- 每个 Agent 的边界由其**职责声明**定义，不由其代码目录定义

## 4. 运行时系统角色

### 4.1 Coordinator / Primary（协调与总答复）

- **职责**：重述问题、识别问题类型、判断是否需要专家、调度专家、汇总意见、组织最终答复
- **边界**：唯一直接对用户输出最终回答的角色；不做专业判断，不做记忆管理
- **入口**：`src/agents/primary_agent.py`

### 4.2 Memory Governor（记忆治理）

- **职责**：
  1. **读取**：根据当前问题取最相关的历史上下文
  2. **组织**：形成当前咨询所需的 dossier（患者档案摘要）
  3. **写回**：把本轮咨询中值得保留的信息写回存储
  4. **冲突处理**：不盲目覆盖旧信息，记录冲突关系
  5. **新鲜度管理**：过期、无效、未确认的信息弱化处理
- **边界**：不做路由决策，不替代其他 Agent 思考，不是"万能总控大脑"
- **底座**：Memory Palace 独立记忆服务（SQLite + 混合检索）
- **入口**：`src/memory/`

### 4.3 Specialist Agents（专家组）

| 专家 | 代号 | 职责 | 入口 |
|------|------|------|------|
| 药学专家 | Pharmacist | 用药方案、药物依从性、药物相互作用与风险提示 | `src/agents/pharmacist_agent.py` |
| 营养/生活方式专家 | Nutritionist | 饮食建议、运动建议、作息与生活方式干预 | `src/agents/nutritionist_agent.py` |
| 医疗安全专家 | Doctor / Safety | 风险识别、是否需要就医/复诊/升级人工、安全把关 | `src/agents/doctor_agent.py` |

### 4.4 Safety Reviewer（安全审查）

- **职责**：对最终答复进行安全检查，拦截可能的医疗风险
- **边界**：只做"是否安全"的判断，不修改答复内容本身
- **当前状态**：逻辑集成在 Coordinator 流程中，未独立为单独 Agent

### 4.5 Evaluation Pipeline（评价管线）

- **职责**：收集咨询日志、管理人工评价标注、分析失败类型、生成优化建议
- **边界**：属于离线优化环，不参与在线服务
- **入口**：`src/evolution/`

## 5. Agent 分工表

### 核心角色（第一阶段必须上线）

| 角色 | 代号 | 状态 | 职责 |
|------|------|------|------|
| 协调 Agent | Primary / Coordinator | 代码+文档 | 重述问题、判断是否需要专家、汇总意见、组织最终答复。唯一直接对用户输出最终回答的角色 |
| 记忆治理 Agent | Memory Governor | 代码+文档 | 读取患者相关记忆、组织 query-specific context、写回新信息、处理冲突/过期/未确认信息 |
| 药学 Agent | Pharmacist | 代码+文档 | 用药方案、药物依从性、药物相互作用与风险提示 |
| 营养/生活方式 Agent | Nutritionist | 代码+文档 | 饮食建议、运动建议、作息与生活方式干预 |
| 医疗安全 Agent | Doctor / Safety | 代码+文档 | 风险识别、是否需要就医/复诊/升级人工、最终安全把关 |

### 扩展角色（非第一阶段）

| 角色 | 代号 | 状态 | 说明 |
|------|------|------|------|
| 科普老师 | Educator | 仅文档 | 健康教育，后续阶段扩展 |
| 心理咨询师 | Counselor | 仅文档 | 心理支持，后续阶段扩展 |

### 分工铁律

- Memory Governor 不是"万能总控大脑"，不负责路由决策或代替其他 Agent 思考
- Coordinator 是唯一的用户交互出口，其他 Agent 不直接对用户发言
- 专家 Agent 输出结构化意见，不做最终综合
- Safety Reviewer 只审查，不改写

## 10. 证据分层标准

在写代码、文档或汇报时，**必须遵循四级分层**：

| 等级 | 含义 | 可以写成 | 典型依据 |
|------|------|---------|---------|
| L1 代码已证实 | 代码仓库有对应实现且可运行 | "已有源码原型" | 代码文件 + 运行日志 |
| L2 文档已明确 | 主稿/流程图有明确设计表达 | "当前设计/当前方案" | 设计文档 + 流程图 |
| L3 设计级细节 | 有具体参数/方案但未实现 | "当前设计方案/配置口径" | 讨论记录 + 设计稿 |
| L4 待复核 | 需回任务书或与导师确认 | "待复核" | 无 |

### 绝对禁止的表述

以下措辞在任何代码注释、文档、汇报中**一律不准出现**：

- "已稳定运行"
- "已完整实现"
- "已跑通闭环"
- "已验证效果显著提升"
- "系统已能自动自进化"

## 13. 明确禁止事项

以下行为在任何开发活动中**明确禁止**：

### 架构与设计禁令

1. **禁止重新引入"三层记忆"作为主线架构** —— 记忆系统是"统一底座 + 治理"
2. **禁止实现无约束自进化** —— 所有优化必须经过人工评价和离线验证
3. **禁止让 Memory Governor 承担路由或协调职责** —— 它只管记忆
4. **禁止让非 Coordinator 角色直接对用户输出最终答复**
5. **禁止在在线服务环中执行优化/修改 prompt 的操作** —— 优化只在离线环发生

### 实现禁令

6. **禁止绕过安全审查流程输出答复**
7. **禁止在代码中硬编码医疗建议内容**（所有医疗建议必须由 LLM 生成并经安全审查）
8. **禁止修改安全红线、医疗高风险规则** —— 这些只能由人类专家修改
9. **禁止在没有测试验证的情况下合入优化方案**

### 文档与口径禁令

10. **禁止使用第 10 节中列出的绝对禁止表述**
11. **禁止把设计级细节（L3）表述为代码已证实（L1）**
12. **禁止在汇报中承诺未实现的功能为"已完成"**

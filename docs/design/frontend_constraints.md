# 前端开发约束文档

> 适用范围：`app/medical-ui/` 全部前端代码
> 版本：v1.0 | 日期：2026-04-09
> 状态：**生效中**

---

## 1. 前端定位

前端**不是"聊天壳子"**。它必须同时承担三种不可替代的价值：

| 价值维度 | 目标用户 | 必须做到 |
|----------|----------|----------|
| **让患者理解系统建议** | 患者本人 | 患者能看懂建议内容、区分普通建议与风险提醒、知道何时需要线下就医 |
| **让内部人员理解系统怎么得出建议** | 研发/产品/临床顾问 | 能看到本轮调用了哪些角色、使用了什么记忆上下文、各专家给出了什么意见、最终汇总逻辑是什么 |
| **让评测人员高效检查失败点** | 评测/质控人员 | 能快速定位问题咨询、打分打标签、标注失败分类、对比不同版本行为差异 |

如果某个界面只服务了其中一种价值，说明设计不完整。如果某个界面让三类用户都觉得"看不明白"，说明设计失败。

## 2. 至少要覆盖的产品界面

### A. 患者咨询界面

**核心职责**：患者与多智能体系统的唯一交互窗口。

必须包含的功能区域：

- **对话主体** — 消息气泡、Markdown 渲染、用户/系统角色区分
- **对话历史** — 可回溯之前的咨询记录，不因刷新丢失
- **答复展示** — 最终综合建议清晰呈现，专家意见可展开查看
- **上下文摘要** — 当前患者的基本信息、近期血糖趋势在对话界面可见，不需要反复切页面
- **风险提示** — 高风险建议（如低血糖警告、需要就医）必须在视觉上突出，不能淹没在普通文本中
- **最近建议回顾** — 之前几轮对话的核心建议可快速查看，不用翻几十条消息

### B. 个案详情 / Trace 审阅界面

**核心职责**：让内部人员完整理解"系统为什么这么回答"。

必须包含的信息：

- **本轮调用了哪些角色** — 主治医生、药剂师、营养师、代谢病医生分别是否参与
- **使用了哪些信息** — 从记忆中检索到了什么上下文、患者档案中提取了什么字段
- **各专家意见** — 每位专家的原始回复，不是摘要而是完整内容
- **汇总过程** — 主治医生如何综合多方意见形成最终答复
- **不确定与冲突** — 如果专家意见互相矛盾或存在不确定性，必须标出

### C. 记忆审阅界面

**核心职责**：让人类检查和管理系统的记忆质量。

必须包含的功能：

- **检索到的上下文** — 某次咨询中实际被检索到的记忆条目
- **新鲜/过期/冲突/待验证标记** — 每条记忆的生命状态，不能全部长得一样
- **写回候选** — 系统打算新增或更新的记忆条目，待人工确认
- **人类快速判断** — 一键确认/拒绝/标记待讨论，不需要填写长表单

### D. 评测界面

**核心职责**：让评测人员高效完成质量评估闭环。

必须包含的功能：

- **查看咨询记录** — 完整的问答对，含患者问题和系统回复
- **打分打标签** — 好/坏/中立/错误，安全性、个性化、建议方向
- **失败分类备注** — 自由文本备注，标注具体失败原因
- **版本行为对比** — 同一问题在不同 prompt 版本或记忆版本下的回答差异（至少预留位置）

### E. 运维/管理界面

**核心职责**：让运维和管理人员掌握系统运行状况。

必须包含的信息：

- **运行情况** — 各服务健康状态、API 响应时间、SSE 连接状态
- **路由统计** — 咨询被路由到哪些专家、各角色参与频率
- **错误情况** — 最近的错误列表、错误分类、影响范围
- **待评测队列** — 尚未被评测的咨询数量和队列深度
- **最近版本变化** — prompt 版本、记忆规则变更记录

## 3. 视觉与美观性要求（硬要求）

### 3.1 总体感受目标

界面必须让使用者感受到：

> **稳定、清晰、专业、医疗场景可信、不像学生作业、不像炫技 demo**

这不是装饰性要求，而是产品可信度的基础。在医疗语境下，视觉粗糙 = 不可信。

### 3.2 应当追求

| 维度 | 具体要求 |
|------|----------|
| **层级结构** | 页面有明确的信息层级，标题 > 区块 > 内容 > 辅助信息，一眼能分清主次 |
| **留白** | 内容区块之间有充足的呼吸空间，不拥挤，不贴边 |
| **字体可读性** | 正文 14px 起步，行高 1.5+，中文字体回退链完整（system-ui → 思源黑体 → sans-serif） |
| **状态颜色语义** | 绿色=正常/安全，黄色=警告/注意，红色=危险/错误，蓝色=信息/进行中，紫色=评测/AI 相关 |
| **卡片面板** | 内容用卡片或面板组织，边界清晰但不生硬，当前已有 `glass-panel` 基础样式可复用 |
| **交互反馈** | 按钮点击有响应、加载有骨架屏或 spinner、提交有成功/失败提示 |
| **重要内容突出** | 风险提醒、异常状态、关键数值必须在视觉上第一时间被注意到 |

### 3.3 应当避免

| 反模式 | 说明 |
|--------|------|
| **花哨渐变和过度装饰** | 不要为了"好看"加太多渐变、光效、粒子。当前 MemoryStarMap 的星图视觉可保留但不要蔓延到其他页面 |
| **过多鲜艳颜色** | 主色调控制在 2-3 个（primary indigo + accent cyan + 语义色），不要每个组件一个颜色 |
| **过多边框/分割线/阴影堆叠** | 用留白和背景色差替代分割线，`glass-panel` 的 `border-white/5` 已经够了，不要再叠加 |
| **密密麻麻的长文本** | 超过 3 行的文本必须有展开/收起，表格内容超过可视区域必须有滚动 |
| **患者界面与内部审阅界面风格混乱** | 患者看到的界面应该温和友好，内部审阅界面可以信息密度更高，但两者不能混在同一个页面里互相干扰 |
| **高风险提醒视觉不突出** | 低血糖警告不能和普通建议用同样的样式，"需要就医"不能用灰色小字 |

### 3.4 当前设计系统资产

已建立的可复用资产（在 `tailwind.config.ts` 和 `globals.css` 中定义）：

```
颜色体系：primary (indigo)、accent (cyan)、warn (amber)、surface (slate)
组件类：glass-panel、glass-panel-hover、btn-primary、btn-ghost
状态标签：badge-good、badge-bad、badge-neutral、badge-error
圆角规范：card=12px、btn=8px
动效：pulse-slow、float
字体栈：system-ui 系统字体链
滚动条：自定义深色滚动条
```

新增组件和页面必须复用这些资产，不得自行发明配色或圆角规范。如需扩展，应先修改 `tailwind.config.ts` 和 `globals.css`，不在组件内部硬编码。

## 4. 交互状态必须完整

每个涉及异步操作或多步骤流程的界面，必须覆盖以下全部状态。缺少任何一个都算实现不完整：

| 状态 | 说明 | 视觉表达要求 |
|------|------|-------------|
| **初始空状态** | 用户首次进入、没有数据 | 友好引导文案 + 操作提示，不能是空白页面。当前 ChatPage 的空状态引导已做到，可作参考 |
| **正在处理中** | 用户提交了请求，系统正在处理 | 加载 spinner 或骨架屏，禁用重复提交按钮 |
| **正在调专家** | 多智能体协作的中间阶段 | 显示当前活跃的角色、已收到的意见数量。当前 `ConsultationStatus` 组件已实现基础版本 |
| **部分失败但可继续** | 某位专家超时或出错，但其他结果可用 | 明确告知哪部分失败、最终建议是否受到影响。不能静默忽略 |
| **最终成功** | 系统完成处理并返回结果 | 结果清晰展示，专家意见可展开，风险提示突出 |
| **需要升级人工医生** | 系统判断当前问题超出 AI 能力范围 | 醒目提示 + 明确的下一步指引（如"建议您联系主治医生"），颜色用 warn 或 error 级别 |
| **无足够信息** | 系统缺少必要的患者信息或记忆 | 引导患者补充信息，说明缺少什么，不能返回模糊的"暂时无法回答" |
| **系统错误** | 服务不可用、网络断开、超时 | 错误信息可理解、有重试按钮、有降级说明。当前 `ErrorToast` 和 `SystemStatusBanner` 已覆盖基础场景 |
| **待人工评测** | 某次咨询结果等待评测人员打分 | 评测页面中有明确的"待评测"标记和计数 |

### 状态转换规则

```
初始空状态 ──[用户提交]──→ 正在处理中
正在处理中 ──[开始调度]──→ 正在调专家
正在调专家 ──[全部完成]──→ 最终成功
正在调专家 ──[部分失败]──→ 部分失败但可继续 ──→ 最终成功（标注降级）
正在调专家 ──[全部失败]──→ 系统错误
最终成功   ──[检测到风险]──→ 需要升级人工医生
任何阶段   ──[超时/断连]──→ 系统错误
任何阶段   ──[信息不足]──→ 无足够信息
最终成功   ──────────────→ 待人工评测（自动进入评测队列）
```

## 6. 验收标准

以下六条验收标准适用于前端整体。任何一条不通过，前端都不算完成。

### 标准一：第一次使用的人能理解主要流程

- 新用户打开页面后，不需要阅读说明文档即可完成一次完整咨询
- 导航结构（侧边栏）清晰，每个入口的用途可以从图标+文字推断
- 对话界面有引导提示（快捷问题按钮），不会让用户面对空白输入框不知所措
- **验证方式**：找一个从未见过系统的人，不给任何提示，观察能否在 2 分钟内完成首次咨询

### 标准二：患者能区分建议、提醒、风险、需升级

- 普通建议、注意事项提醒、高风险警告、需要线下就医在视觉上有明确不同的样式
- 高风险信息不能被普通内容淹没，在滚动时也能快速定位
- 患者不需要仔细阅读全文才能发现"需要去看医生"这样的关键信息
- **验证方式**：展示一组混合了普通建议和高风险提醒的回复，让测试者指出哪些需要特别注意

### 标准三：审阅者能看懂系统为何这么答

- 审阅者可以看到完整的决策链：记忆检索 → 专家调度 → 各专家意见 → 综合汇总
- 当某位专家没有参与时，能看到原因（未被调度、超时、出错）
- 当专家意见存在冲突时，审阅者能发现并理解系统如何处理
- **验证方式**：对一次咨询，审阅者能在 30 秒内说出"系统调了哪几个专家、用了什么记忆、为什么给出这个建议"

### 标准四：重要状态都有明确视觉表达

- 第 4 节列出的 9 种交互状态，每种都有对应的 UI 表达
- 不存在"系统在后台处理但用户界面没有任何变化"的情况
- 错误状态不会让用户困惑（"到底出了什么问题？""我该怎么办？"）
- **验证方式**：逐一模拟每种状态，确认 UI 有正确反馈且用户能理解当前发生了什么

### 标准五：界面风格统一不显得临时拼凑

- 所有页面使用统一的颜色体系、字体规范、间距规范、卡片样式
- 不同页面之间切换不会有"这像是两个不同的产品"的感觉
- 自定义组件都基于 `globals.css` 中定义的基础类构建
- **验证方式**：截图所有页面并排放置，视觉风格应该一致

### 标准六：在医疗语境下让人感到可信而不是玩具感

- 界面的专业程度至少不低于市面上主流的健康管理 App
- 不会出现明显的对齐错乱、颜色刺眼、文字溢出、组件重叠等低级问题
- 深色主题在医疗场景下的配色让人感到沉稳专业，不是游戏界面
- 加载和过渡动效克制得体，不过度动画
- **验证方式**：让一位非技术背景的医疗从业者看界面截图，问"你觉得这是正式产品还是学生作业？"

## 附录：开发约定速查

### 技术约定

- **状态管理**：Zustand，按领域拆 store（chat / evaluation / memory / patient），不引入 Redux
- **样式方案**：Tailwind utility classes + `globals.css` 中的 `@layer components` 组件类，不写 CSS modules
- **动效**：framer-motion，仅用于有意义的状态转换（出现/消失/切换），不做纯装饰动画
- **API 通信**：`api/client.ts`（REST）+ `api/ccccClient.ts`（CCCC SSE/ledger），不直接用 fetch
- **路由**：react-router-dom v6，新页面加到 `App.tsx` 的 `<Routes>` 中

### 文件组织约定

```
src/
  pages/          # 页面级组件，对应路由
  components/     # 按领域组织的组件目录
    chat/         # 对话相关组件
    common/       # 跨领域通用组件
    evaluation/   # 评测相关组件
    layout/       # 布局组件（AppShell, Sidebar, Header）
    memory/       # 记忆相关组件
    profile/      # 患者档案相关组件
  stores/         # Zustand store，按领域一个文件
  api/            # API 客户端
  styles/         # 全局样式
```

新增界面（如 Trace 审阅、运维面板）应遵循同样的目录结构：在 `components/` 下新建对应子目录，在 `pages/` 下新建页面组件，在 `stores/` 下新建 store（如有需要）。

### 颜色语义速查

| 颜色 | 语义 | Tailwind token |
|------|------|---------------|
| Indigo (primary) | 主品牌色、主要操作、选中态 | `primary-*` |
| Cyan (accent) | 辅助信息、次要操作 | `accent-*` |
| Emerald | 正常/安全/好评 | `emerald-*` |
| Amber | 警告/注意 | `amber-*` / `warn-*` |
| Red | 危险/错误/差评 | `red-*` |
| Purple | AI/评测相关 | `purple-*` |
| Gray/Slate | 次要文本、边框、背景层级 | `gray-*` / `surface-*` |

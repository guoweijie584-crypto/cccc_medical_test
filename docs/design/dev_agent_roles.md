## 1. System Architect Agent

### 负责

- 全局架构一致性
- 模块责任划分
- 在线环 / 离线环边界
- 重要设计决策说明
- 风险记录

### 交付物

| # | 交付物 | 说明 |
|---|--------|------|
| 1 | 系统架构图或等价描述 | 描述系统整体结构及模块关系 |
| 2 | 角色责任表 | 各模块/代理的责任边界 |
| 3 | 模块依赖图 | 模块间调用与数据依赖关系 |
| 4 | 设计决策记录（ADR） | Architecture Decision Records，记录关键决策及其理由 |
| 5 | 风险清单 | 已识别的技术风险、架构风险及缓解措施 |

### 不得

- 越权细化所有 UI 细节
- 越权改写数据规范但不通知数据负责人
- 把"理想架构"冒充成"当前已完成实现"

## 2. Frontend Experience Agent

### 负责

- 患者端界面
- 评审端界面
- 管理 / 运维界面
- 可读性、可用性与视觉一致性
- 重要状态的可视化表达

### 交付物

| # | 交付物 | 说明 |
|---|--------|------|
| 1 | 页面结构图 | 各页面的层级与导航关系 |
| 2 | 关键交互流程图 | 核心用户操作路径 |
| 3 | 组件清单 | 可复用组件列表及用途 |
| 4 | UI 状态规范 | 各组件/页面的状态定义（loading、error、empty 等） |
| 5 | 前端验收说明 | 验收标准与测试方法 |

### 不得

- 假设后台一定会提供未约定的数据
- 自己定义核心业务规则
- 为了炫酷牺牲医疗场景可信感

## 3. Backend Orchestration Agent

### 负责

- 在线咨询工作流
- 动态路由
- 专家调用编排
- 安全审查接入
- trace 生成与记录
- mock/real 模式边界

### 交付物

| # | 交付物 | 说明 |
|---|--------|------|
| 1 | 工作流说明 | 各业务流程的状态机或流程图 |
| 2 | 输入输出契约 | 各接口的请求/响应结构定义 |
| 3 | 异常处理策略 | 错误码、fallback 路径、重试规则 |
| 4 | 日志与可观测事件规范 | trace/span/metric 定义及采集规则 |
| 5 | 后端验收说明 | 验收标准与测试方法 |

### 不得

- 把核心策略藏在不可审查的 prompt 里
- 隐藏 fallback 行为
- 把临时测试逻辑混成正式业务逻辑

## 4. Memory & Data Governance Agent

### 负责

- 记忆类型设计
- 记忆元数据设计
- 检索与写回规则
- 冲突 / 新鲜度 / 验证状态处理
- 数据存储与审计约束

### 交付物

| # | 交付物 | 说明 |
|---|--------|------|
| 1 | Memory Schema / Entity Taxonomy | 记忆实体类型定义及层级关系 |
| 2 | Metadata 规范 | 元数据字段、取值范围、必填/选填规则 |
| 3 | 检索约束 | 检索条件、排序规则、过滤策略 |
| 4 | 写回约束 | 写入条件、冲突处理、版本控制规则 |
| 5 | 审计与历史可追踪方案 | 变更日志、审计链路、数据保留策略 |

### 不得

- 把 memory 变成无结构文本垃圾桶
- 无提示覆盖冲突信息
- 擅自接管路由与最终答复责任

## 5. Evaluation & Quality Agent

### 负责

- 人类评测流程
- Failure taxonomy
- 回归集与对比机制
- 上线门槛建议
- 验收证据收集

### 交付物

| # | 交付物 | 说明 |
|---|--------|------|
| 1 | 评测 Rubric | 评分维度、评分标准、示例 |
| 2 | 失败分类表 | Failure taxonomy，含严重程度与归因 |
| 3 | Benchmark Case 结构 | 基准测试用例格式与管理规则 |
| 4 | 回归验证规则 | 版本间回归检测方法与通过标准 |
| 5 | 版本对比报告模板 | 版本间质量对比的标准报告格式 |

### 不得

- 用模糊描述替代评价标准
- 因为个别样例看起来不错就宣称系统变好了
- 跳过回归检查直接批准改动

## 6. Documentation & Handoff Agent

### 负责

- 文档持续更新
- 模块说明整理
- 限制与风险透明表达
- 面向人类的交接材料

### 交付物

| # | 交付物 | 说明 |
|---|--------|------|
| 1 | 当前状态说明 | 系统整体当前状态的准确描述 |
| 2 | 模块文档 | 各模块的功能、接口、限制说明 |
| 3 | Changelog / 版本记录 | 变更历史，含新增、修改、删除 |
| 4 | 未解决问题列表 | 已知问题、待定事项、开放讨论 |
| 5 | 人类审查清单 | 供人类审查者使用的检查项 |

### 不得

- 把未来规划写成已完成能力
- 混淆"设计中""实现中""已完成"
- 省略限制条件与不确定性

---

## 7. Integration Lead Agent

### 负责

- 合并各开发代理成果
- 检查接口契约是否一致
- 组织端到端联调
- 检查验收证据是否齐全

### 交付物

| # | 交付物 | 说明 |
|---|--------|------|
| 1 | 集成检查单 | 各模块集成前的检查项 |
| 2 | 接口错配清单 | 已发现的接口不一致问题及状态 |
| 3 | 端到端 Readiness 报告 | 联调就绪度评估报告 |
| 4 | 依赖问题汇总 | 跨模块依赖问题及解决进展 |

### 不得

- 为了先跑起来就绕开契约与验收
- 私自填补缺失逻辑而不记录
- 在关键模块不一致时强行合并并宣称完成

## 协作协议

### 开始前声明（每个代理必须）

每个代理在开始工作前，必须写清楚以下内容：

1. **负责什么** — 本次工作的具体范围
2. **不负责什么** — 明确排除的范围
3. **依赖哪些输入** — 需要从哪些代理/来源获取什么数据或文档
4. **输出给谁** — 交付物的接收方
5. **交付物怎样验收** — 验收标准与验收方式
6. **预期哪些风险** — 已识别的风险及缓解措施

### 交付时附带（每次交付必须）

每次交付必须附带以下信息：

1. **做了什么** — 本次交付的具体内容
2. **没做什么** — 明确声明未完成的部分
3. **依赖了什么假设** — 本次工作基于哪些假设
4. **如何验证** — 验证方法与验证结果
5. **还有哪些开放问题** — 待解决的问题与讨论点

### 不允许的行为

- 多个代理未对齐契约时，同时修改同一责任边界
- 先写代码后补约束文档
- 前端假设后台有 / 后台假设前端懂
- 上线逻辑没有对应可观测信息

## 推荐交付模板

### 15.1 开始前声明模板

```markdown
## 开始前声明

- **代理角色**：[角色名称]
- **任务编号**：[任务ID]
- **日期**：[YYYY-MM-DD]

### 负责什么
- [具体范围 1]
- [具体范围 2]

### 不负责什么
- [排除范围 1]
- [排除范围 2]

### 依赖输入
| 来源代理 | 所需内容 | 状态 |
|----------|---------|------|
| [代理名] | [内容描述] | 已获取 / 待确认 / 缺失 |

### 输出给谁
| 接收代理 | 交付内容 | 预计时间 |
|----------|---------|---------|
| [代理名] | [内容描述] | [时间] |

### 验收标准
- [ ] [验收项 1]
- [ ] [验收项 2]

### 预期风险
| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| [风险描述] | [影响程度] | [缓解方法] |
```

## 一、总体概述

### 1.1 产品定位

一个**面向用户友好的血糖管理智能助手产品**。用户（糖尿病患者）通过与 AI Agent 对话来获取血糖管理建议，系统在后台通过多 Agent 协作、记忆管理、人工评价驱动的自进化来持续提升服务质量。

### 1.2 用户角色

| 角色 | 说明 | 核心功能 |
|------|------|---------|
| **患者** | 产品的主要使用者 | 和 Agent 对话咨询、查看个人健康档案 |
| **医生/评审员** | 对系统回答进行评价 | 评价管理、评价统计 |
| **管理员** | 系统管理者 | 记忆管理、系统监控（可选） |

### 1.3 技术架构

```
┌───────────────────────────────────────────────────────┐
│             独立前端 (medical-ui/)                      │
│             Vite + React 18 + TypeScript               │
│             Tailwind CSS + framer-motion               │
│             端口 :5173 (dev) / nginx 代理              │
├───────────────────────────────────────────────────────┤
│                        ↕ HTTP API                      │
├───────────────────────────────────────────────────────┤
│             API Server (:8001)                         │
│             FastAPI + Python                           │
├──────────────────┬────────────────────────────────────┤
│                  ↕                                     │
│  Memory Palace (:8000)  |  Multi-Agent Workflow       │
│  记忆存储与治理          |  多Agent协作生成回答         │
└──────────────────┴────────────────────────────────────┘
```

## 二、模块一：Chat 对话（核心功能）

### 2.1 功能概述

用户通过对话界面与血糖管理 AI Agent 进行实时交流。这是产品的**核心入口**，用户打开产品后默认进入此页面。

### 2.2 页面布局

```
┌─────────────────────────────────────────────────────────────┐
│  🏥 血糖管理智能助手              患者：张三 · 58岁 · 2型   │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌─ Agent ──────────────────────────────────────────────┐   │
│  │ 您好！我是您的血糖管理助手。根据您的健康档案，您最近   │   │
│  │ 的空腹血糖控制在 7.2 mmol/L 左右，稍有偏高。          │   │
│  │ 有什么我可以帮您的吗？                                 │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│                    ┌─ 用户 ─────────────────────────────┐   │
│                    │ 我最近血糖一直偏高，该怎么调整？     │   │
│                    └────────────────────────────────────┘   │
│                                                              │

│  │ 🥗 营养师建议：减少高GI食物摄入...                    │   │

│  │                                                       │   │
│  │ 综合建议：建议您先调整饮食结构，同时预约一次...        │   │

│  [输入您的健康问题...]                          [📎] [发送] │
└──────────────────────────────────────────────────────────────┘
```

### 2.3 功能需求

#### 2.3.1 对话消息展示
- 支持用户消息（右侧气泡）和 Agent 消息（左侧气泡）
- Agent 回答需展示多专家意见的整合结果（药剂师/营养师/代谢医生各自建议 + 主治医生综合建议）
- 支持 Markdown 渲染（Agent 回答中的列表、加粗、代码块等）
- 消息带时间戳

#### 2.3.2 消息输入
- 底部固定的输入框
- 支持回车发送、按钮发送
- 发送后输入框清空
- 发送中禁用输入（防重复提交）

#### 2.3.3 Agent 状态
- Agent 处理中显示"正在思考"动画（脉动圆点或类似效果）
- 多 Agent 协作过程对用户透明（用户只看到最终整合结果）

#### 2.3.4 对话上下文
- 系统自动读取患者的 Memory Palace 记忆作为上下文
- 对话历史在会话期间保持
- 每轮对话完成后，系统自动提取事实并写入 Memory Palace
- 每轮对话完成后，系统自动创建待评价记录

### 2.4 对接的后端 API

| API | 方法 | 用途 |
|-----|------|------|
| `/api/consultation` | POST | 发起咨询（发送消息） |
| `/api/patients/{id}` | GET | 获取当前患者信息 |
| `/api/patients/{id}/memories` | GET | 获取患者记忆（用于展示上下文） |

### 2.5 视觉与交互要求

- **深色主题为主**，医疗科技感
- 消息气泡有柔和阴影和圆角
- Agent "思考中" 动画要流畅、有科技感
- 多专家意见区域用不同颜色的图标和分隔线区分
- 页面打开时有平滑的加载过渡动画

## 三、模块二：记忆宫殿（Memory Palace）

### 3.1 功能概述

以可视化方式展示和管理患者在 Memory Palace 中的所有健康记忆。这是本产品最核心的**视觉亮点**，需要做到"上帝管理记忆的感觉"，让评委眼前一亮。

### 3.2 页面布局

```
┌──────────────────────────────────────────────────────────────┐
│  🏛️ 记忆宫殿                                    [搜索🔍]    │
├────────────┬─────────────────────────────────────────────────┤
│            │                                                 │
│ 患者选择    │        记忆星图（Canvas 全屏渲染）               │
│            │                                                 │
│ ● 张三     │     ★ profile (金色大星)                        │
│ ○ 李四     │    /  |  \                                      │
│ ○ 王五     │  ★   ★    ★                                    │
│            │ glucose medication diet                         │
│            │ (青色)  (蓝色)  (绿色)                           │
│            │                                                 │
│ [+ 新建]   │   能量粒子在连接线上流动...                      │
│            │   星尘粒子漂浮在深空背景中...                     │
│            │                                                 │
│            ├─────────────────────────────────────────────────┤
│            │  📋 记忆详情（选中节点后展开）                    │
│            │  类别：血糖   优先级：2   活力：85%               │
│            │  内容：空腹血糖 7.2 mmol/L                       │
│            │  触发条件：当评估血糖控制情况时                   │
│            │  [编辑] [删除]                                   │
└────────────┴─────────────────────────────────────────────────┘
```

### 3.3 功能需求

#### 3.3.1 记忆星图可视化（Canvas 渲染，已有基础实现）
- **深空背景**：深蓝/紫渐变 + 120+ 颗实时闪烁的星尘粒子
- **中心枢纽**：紫色脉动光球，代表患者的记忆核心
- **记忆节点**：3D 感发光球体，径向渐变模拟球体光泽
  - 按类别着色：profile=金色, glucose=青色, medication=蓝色, diet=绿色, safety=红色, complication=橙色, consultation=紫色
  - 按 priority 决定大小（priority 0 = 最大）
  - 按 vitality 决定脉搏频率和亮度（高活力=快速明亮脉搏）
- **能量脉络**：中心到节点的连线，带有流动的发光粒子
- **交互**：
  - 悬停节点：显示 tooltip（类别/优先级/活力/触发条件）
  - 点击节点：打开详情面板
  - 搜索高亮：匹配节点高亮闪烁，不匹配节点暗淡
- **中文类别标签**：环绕在星图边缘（档案/血糖/用药/饮食/运动/安全/并发症/咨询）

#### 3.3.2 记忆管理操作
- **搜索**：输入关键词搜索记忆，350ms 防抖
- **查看详情**：选中节点后展示完整内容、元数据
- **新建记忆**：弹窗表单，支持类别选择、优先级、触发条件
- **编辑记忆**：修改内容、优先级、触发条件
- **删除记忆**：确认后删除

#### 3.3.3 患者切换
- 左侧患者列表
- 切换患者后重新加载该患者的记忆树

### 3.4 对接的后端 API

| API | 方法 | 用途 |
|-----|------|------|
| `/api/patients` | GET | 获取患者列表 |
| `/api/memory/tree/{patient_id}` | GET | 获取记忆树（递归 3 层） |
| `/api/memory/search?q=&patient_id=` | GET | 搜索记忆 |
| `/api/memory/create` | POST | 创建记忆 |
| `/api/memory/{path}` | PUT | 更新记忆 |
| `/api/memory/{path}` | DELETE | 删除记忆 |
| `/api/memory/stats?patient_id=` | GET | 记忆统计 |

### 3.5 视觉与交互要求

- **这是全产品的视觉高光**，必须做到惊艳效果
- Canvas 渲染，60fps 流畅动画
- 深空主题与全产品的深色主题统一
- 节点出现/消失有凝聚/碎裂动画（framer-motion 配合 Canvas）
- tooltip 和详情面板有毛玻璃效果

供医生/评审员对系统的回答质量进行人工评价。评价数据驱动系统的自进化优化（线3）。

### 4.2 页面布局

```

│  │ 好评率: 65%                    需关注: 20             │   │
│  └──────────────────────────────────────────────────────┘   │
│                                                              │
│  ┌── 待评价 ────────────────────────────────────────────┐   │
│  │                                                       │   │
│  │  [患者P001] "我的血糖控制得怎么样？"                  │   │
│  │  系统回答: "根据您近期的血糖记录..."                   │   │
│  │  [展开查看完整回答和专家意见]                          │   │
│  │                                                       │   │
│  │  [👍 好]  [👎 坏]  [➖ 中立]  [⚠️ 错误]              │   │
│  │                                                       │   │
│  │  ▸ 安全性: [安全] [有风险] [危险]    (可选)           │   │

│  └──────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────┘
```

### 4.3 功能需求

#### 4.3.1 统计概览
- 8 张统计卡片：总计/好/坏/中立/错误/好评率/待评价数/需关注数
- 支持按患者筛选
- 数据实时刷新（提交评价后自动更新）

#### 4.3.2 待评价列表
- 展示所有待评价记录
- 每条记录显示：患者、原始问题、系统回答摘要
- 可展开查看完整回答和各专家意见
- 四个标签按钮：**好(GOOD) / 坏(BAD) / 中立(NEUTRAL) / 错误(ERROR)**
- 可选扩展字段：安全性、建议方向、备注
- 提交后该条记录从待评价列表消失，统计自动更新

#### 4.3.3 历史评价
- 展示所有已评价记录
- 支持按标签类型筛选（全部/好/坏/中立/错误）
- 每条显示标签 badge、患者、问题、时间
- 支持查看详情

### 4.4 对接的后端 API

| API | 方法 | 用途 |
|-----|------|------|
| `/api/evaluations/pending` | GET | 获取待评价列表 |
| `/api/evaluations/pending` | POST | 创建待评价（系统自动调用） |
| `/api/evaluations/{id}/submit` | POST | 提交评价 |
| `/api/evaluations/stats` | GET | 评价统计 |
| `/api/evaluations/bad` | GET | BAD/ERROR 评价列表 |
| `/api/evaluations/{id}` | GET | 单条评价详情 |

### 4.5 视觉与交互要求

- 简洁专业的医疗系统后台风格
- 标签按钮大而醒目，方便快速操作
- 统计卡片有 loading skeleton 动画
- 提交后有成功反馈动画

const API_BASE = import.meta.env.VITE_API_BASE || '/api';

async function apiRequest<T>(path: string, options?: RequestInit): Promise<T> {

# Definition of Done — 血糖管理多智能体系统验收清单

> 本文档定义各工作流的"完成"标准、强制验收清单，以及人类角色检查项。
> 所有验收均以"可交付、可追溯、可交接"为底线。

---

## 1. Definition of Done（按工作流分）

### 1.1 Frontend Done

- [ ] 页面对应真实工作流状态——不是静态截图，而是反映实际后端状态机
- [ ] 风险和不确定性表达清楚——系统不确定时用户能感知到，而非被隐藏
- [ ] 患者端与审阅端目标清晰不同——患者端聚焦理解与行动，审阅端聚焦审核与干预
- [ ] 视觉质量达到统一、可信、专业——不允许拼凑感、风格混乱、业余排版
- [ ] 核心组件与状态不是临时拼凑——关键交互有稳定的组件封装，不是散落的 inline hack

### 1.2 Backend Done

- [ ] 端到端流程稳定——从输入到输出，主流程可重复执行且结果一致
- [ ] 关键决策可追踪——每个重要分支点都有日志或元数据记录"为什么走了这条路"
- [ ] 异常路径被显式处理——不允许静默吞错，失败必须有明确的状态码与用户可理解的提示
- [ ] mock 与 real 的边界可说明——哪些模块是真实实现、哪些是桩/模拟，有清晰标注
- [ ] 上层界面能有效消费后端数据——API 返回的结构和语义足以让前端完成渲染，无需猜测

### 1.3 Memory/Data Done

- [ ] 检索和写回有规则——不是"什么都存、随便取"，而是有明确的读写策略
- [ ] memory 不是纯文本垃圾堆——存储内容有结构、有标签、有时效性标记
- [ ] 冲突、过期、验证状态可管理——当记忆条目之间矛盾或过期时，系统能识别并处理
- [ ] 审阅者能理解本轮上下文来源——系统用了哪些记忆来做决策，审阅者可以看到并验证

### 1.4 Evaluation Done

- [ ] 人类专家能高效打分与打标签——评测界面/流程不给评测者增加不必要的负担
- [ ] 失败分类可复用——失败不是一次性描述，而是有类型体系，可跨版本统计
- [ ] 版本之间能比较——同一场景在不同版本下的表现差异可量化、可回溯
- [ ] 团队能基于证据讨论"是否变好"——不靠直觉，靠数据和标注结果做决策

### 1.5 Documentation Done

- [ ] 文档反映当前真实状态——不是三个月前的计划书，而是今天系统的实际样子
- [ ] 文档区分设计目标与已实现能力——写清楚"想做什么"和"做到了什么"，不混淆
- [ ] 文档附带限制与风险——已知缺陷、安全边界、性能瓶颈都要写出来
- [ ] 文档足以支持交接与联调——新成员或外部系统对接时，靠文档就能开始工作

## 2. 强制验收清单

### 2.1 产品层

- [ ] 项目定位是否仍然清楚？——核心价值主张没有漂移
- [ ] 有没有重新发散成"啥都想做"？——功能范围有明确边界，拒绝列表存在
- [ ] 在线服务与离线优化是否仍然清晰分离？——实时响应与批量分析/训练是两条独立路径

### 2.2 架构层

- [ ] 每个模块责任是否清楚？——一句话能说明该模块做什么、不做什么
- [ ] 有没有某个模块悄悄承担过多责任？——没有"上帝模块"把所有逻辑都塞进去
- [ ] 不看代码时团队能否讲清主流程？——用白板或文档就能让新人理解系统如何运行

### 2.3 前端层

- [ ] 患者第一次使用能否理解？——无需培训即可完成核心流程
- [ ] 风险/升级提示够不够明显？——危险血糖值、需要就医等信息不能被淹没
- [ ] 审阅者能否看懂系统为何这么答？——推理依据、引用来源在审阅界面可见
- [ ] UI 是否达到专业可信最低标准？——不允许破碎布局、不一致配色、信息层级混乱

### 2.4 后端层

- [ ] 是否支持端到端真实流程？——从患者提问到返回建议，完整链路可跑通
- [ ] 是否能复盘关键决策？——事后可以还原"系统为什么给出这个建议"
- [ ] 是否显式处理失败而不是吞掉错误？——每个异常都有对应的处理策略，而非 `catch {}` 空块
- [ ] 是否把 mock/real 边界写清楚？——README 或注释中标明哪些模块尚未真实实现

### 2.5 数据/记忆层

- [ ] 记忆检索是否可解释？——能回答"为什么选了这条记忆而不是那条"
- [ ] 写回是否可控？——不会因为一次对话就覆盖核心健康档案
- [ ] 冲突和过期是否可表达？——系统能标记"这两条记忆矛盾"或"这条已过期"
- [ ] 审阅者能否看懂用到的上下文？——上下文不是黑盒，审阅者可以逐条检查

### 2.6 评价层

- [ ] 人类评测是否方便执行？——评测流程的认知负荷低、操作步骤少
- [ ] 失败标签是否可复用？——标签体系有版本管理，跨评测轮次一致
- [ ] 改动是否有比较证据？——每次改动都有 before/after 的量化对比
- [ ] 是否具备"不过线不升级"的机制？——有明确的质量门槛，低于门槛不允许发布

### 2.7 文档层

- [ ] 文档是否真实反映现状？——最后更新时间与系统实际状态匹配
- [ ] 文档是否说清楚了已完成与未完成？——用明确标记区分 `[已实现]` 和 `[规划中]`
- [ ] 文档是否可以直接给新人/AI 代理接手？——无需口头补充即可理解系统全貌

## 3. 人类检查清单

> 以下按 6 个角色分别列出检查项。每个角色在验收时独立执行自己的清单。

### 3.1 人类产品负责人

| # | 检查项 | 通过标准 |
|---|--------|----------|
| 1 | 产品定位核查 | 核心价值主张与立项时一致，未发生静默漂移 |
| 2 | 功能范围审查 | 当前迭代范围有明确边界，存在"不做"列表 |
| 3 | 用户故事覆盖 | 核心用户故事均有对应的可运行功能 |
| 4 | 在线/离线分离 | 实时服务与离线优化路径没有耦合 |
| 5 | 竞品/合规定位 | 产品定位在监管与市场环境中仍然成立 |
| 6 | 交付节奏合理性 | 剩余工作量与时间预算匹配，没有"最后一周做一半"的风险 |

### 3.2 人类医疗/安全审查者

| # | 检查项 | 通过标准 |
|---|--------|----------|
| 1 | 临床准确性 | 血糖管理建议符合当前临床指南（如 ADA 标准） |
| 2 | 风险分级正确性 | 高血糖/低血糖告警阈值和升级路径符合医学共识 |
| 3 | 免责与边界声明 | 系统在合适位置明确声明"不替代医生诊断" |
| 4 | 极端场景处理 | 危险值、矛盾输入、缺失数据等边界情况有安全兜底 |
| 5 | 可追溯性 | 系统给出的每条建议都能追溯到具体的推理依据 |
| 6 | 误导风险评估 | 评估系统在已知失败模式下是否可能产生有害输出 |
| 7 | 升级通道有效性 | 当系统能力不足时，转人工/转医生的通道确实可用 |

### 3.3 人类记忆/数据审查者

| # | 检查项 | 通过标准 |
|---|--------|----------|
| 1 | 记忆结构合理性 | 存储的数据有清晰的 schema，不是自由文本堆积 |
| 2 | 检索策略审查 | 检索逻辑有文档说明，召回结果可解释 |
| 3 | 写回策略审查 | 写入规则清晰，不会意外覆盖或丢失关键信息 |
| 4 | 冲突处理机制 | 矛盾记忆有标记和解决策略，不会静默选择一方 |
| 5 | 过期与时效管理 | 过期数据有标记机制，不会被当作当前事实使用 |
| 6 | 隐私与数据安全 | 敏感健康数据的存储和访问符合隐私要求 |
| 7 | 上下文透明度 | 审阅者可以看到系统在本轮对话中使用了哪些记忆条目 |

### 3.4 人类 UX/UI 审查者

| # | 检查项 | 通过标准 |
|---|--------|----------|
| 1 | 首次使用体验 | 新用户无需培训即可完成核心流程 |
| 2 | 信息层级清晰度 | 重要信息（风险、建议）在视觉上优先展示 |
| 3 | 风险提示可见性 | 告警和升级提示在视觉上足够醒目，不可忽略 |
| 4 | 患者端/审阅端差异 | 两端界面目标不同，信息密度和操作方式有区分 |
| 5 | 视觉一致性 | 配色、字体、间距、组件风格全局统一 |
| 6 | 专业可信度 | 整体视觉质量达到医疗级应用的最低标准 |
| 7 | 响应式与可访问性 | 主要流程在不同设备上可用，基本可访问性达标 |
| 8 | 状态反馈完整性 | 加载、成功、失败、空状态均有明确视觉反馈 |

### 3.5 人类工程/集成审查者

| # | 检查项 | 通过标准 |
|---|--------|----------|
| 1 | 端到端流程可跑通 | 主流程从头到尾可以在测试环境中完整执行 |
| 2 | 模块职责清晰 | 每个模块/服务的边界和接口有文档说明 |
| 3 | 错误处理覆盖 | 主要异常路径有显式处理，无静默吞错 |
| 4 | mock/real 边界标注 | 哪些模块是真实实现、哪些是模拟，有清晰标注 |
| 5 | API 契约一致性 | 前后端之间的接口定义与实际行为一致 |
| 6 | 日志与可观测性 | 关键路径有足够的日志，问题发生时可以定位 |
| 7 | 部署与环境文档 | 部署步骤清晰，环境依赖有说明，新人可独立搭建 |
| 8 | 决策可复盘 | 系统的关键决策点有元数据记录，事后可还原推理过程 |

### 3.6 人类评测负责人

| # | 检查项 | 通过标准 |
|---|--------|----------|
| 1 | 评测流程可执行 | 评测者可以在 30 分钟内完成一轮完整评测 |
| 2 | 评分标准明确 | 打分维度和尺度有书面定义，评测者间一致性可接受 |
| 3 | 失败分类体系 | 失败类型有结构化标签，可跨版本统计和对比 |
| 4 | 版本比较能力 | 同一测试集在不同版本下的结果可并排查看 |
| 5 | 质量门槛定义 | 有明确的"不过线不升级"标准，且团队已达成共识 |
| 6 | 评测数据管理 | 评测结果有版本标记，可回溯到具体的系统版本 |
| 7 | 改进证据链 | 每次系统改动都有 before/after 的量化证据支撑 |
| 8 | 评测工具可用性 | 评测工具本身稳定可用，不会成为评测瓶颈 |

---

## 附：使用说明

1. **每轮迭代结束时**，按本文档逐项检查，未通过项必须记录原因和修复计划。
2. **人类检查清单**由对应角色独立执行，结果汇总后在验收会议上讨论。
3. **Definition of Done** 是交付底线，不是加分项——不满足则不算完成。
4. **本文档本身也需要维护**——当系统架构或产品定位发生变化时，同步更新此文档。

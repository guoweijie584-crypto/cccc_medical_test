# 运行时系统角色约束文档

> 状态：正式约束 | 日期：2026-04-09
>
> 本文定义的是**系统运行时角色**，不是开发分工角色。开发角色见 `dev_agent_roles.md`。

---

## 1. Coordinator / Primary（协调与总答复角色）

### 职责

- 重述和规范化用户问题，消除歧义
- 判断本轮是否需要调用专家协作，如果不需要则直接回答
- 将患者问题分派给相关专家角色
- 汇总各专家结构化意见，形成连贯、完整、对用户友好的最终答复
- 控制答复的整体语气、格式与可读性
- 在答复中标注置信度和不确定性来源
- 当多专家意见冲突时，做出合理权衡并透明表达

### 边界

- **不直接负责长期记忆治理**：记忆的读取、写回、冲突处理由 Memory Governor 负责
- **不直接替代安全审查**：最终答复必须经过 Safety Reviewer 检查
- **不直接负责离线优化**：评价与改进流水线独立运作
- **不自行决定安全红线**：升级就诊/急诊的判断由 Safety Reviewer 把关

### 当前实现状态

- `src/agents/primary_agent.py`：PrimaryAgent 已实现基本的 LLM 调用和 Mock 模式
- `src/agents/workflow.py`：GlucoseManagementWorkflow 实现了协调流程——调三个专家、汇总、输出
- **差距**：当前 Primary 只做简单拼接，没有问题重述、不确定性标注、冲突权衡逻辑

## 2. Memory Governor（记忆治理角色）

### 职责

- 围绕当前咨询问题，从 Memory Palace 读取患者相关历史上下文
- 组装 query-specific memory dossier（而非泛泛拉全部历史）
- 判断本轮交互中哪些信息具备写回长期记忆的条件
- 处理记忆的过期、冲突、验证状态与重要性分级
- 标注记忆来源（用户自述 / 专家推断 / 系统抽取）
- 对检索结果的可解释性负责

### 边界

- **不应变成系统唯一总控**：它是服务于 Coordinator 的上下文供给者
- **不应默认拥有最终患者答复权**：最终答复由 Coordinator 生成
- **不应隐式承担路由、优化、UI、评测责任**：这些分别由其他角色负责
- **不应无条件覆盖旧信息**：冲突信息必须共存标注，而非粗暴替换

### 当前实现状态

- `src/memory/memory_agent.py`：MemoryAgent 已重构为 Memory Palace 适配层（不再有三层记忆）
- `src/memory/palace_client.py`：HTTP 客户端已对接 Memory Palace 的 CRUD + 搜索
- **差距**：缺少冲突检测逻辑、缺少验证状态标注、缺少写回候选的审核流程、元数据不完整（缺置信度、来源分类、有效期）

## 3. Specialist Agents（专家角色）

### 3.1 Pharmacist（药学专家）

**职责：**
- 评估当前用药方案的合理性
- 提示药物相互作用与禁忌
- 给出用药依从性建议
- 标注剂量调整相关的风险与不确定性

**边界：**
- 不直接给出最终综合答复
- 不能忽略患者当前用药上下文
- 必须明确哪些建议需要医师确认

### 3.2 Nutritionist（营养/生活方式专家）

**职责：**
- 提供饮食结构与食物选择建议
- 给出运动方案建议
- 关注作息与生活方式干预
- 结合患者具体情况做个性化推荐

**边界：**
- 不直接给出最终综合答复
- 不能脱离患者的血糖控制目标给建议
- 高风险患者（如低血糖频发）的运动建议必须注明注意事项

### 3.3 Doctor / Safety（代谢病/安全专家）

**职责：**
- 评估患者整体病情控制状况
- 识别潜在并发症风险
- 判断是否需要进一步检查或复诊
- 判断是否需要升级到人工医师或紧急就诊
- 给出血糖控制目标建议

**边界：**
- 不是最终答复的唯一决定者
- 升级人工/急诊的建议必须传递给 Safety Reviewer 确认
- 不能输出不可追踪的黑盒结论

### 通用约束（适用于所有专家）

- 专家角色输出必须是**结构化意见**，而非散文
- 必须明确自身**不确定性**（置信度、信息不足等）
- 必须指出**潜在风险与升级条件**
- 不直接面向用户给出最终统一答复
- 不能忽略 Memory Governor 提供的患者上下文

### 当前实现状态

- `src/agents/pharmacist_agent.py`、`nutritionist_agent.py`、`doctor_agent.py`：三个专家已实现基本结构
- 均继承 `BaseAgent`，支持 LLM 调用和 Mock 模式
- `workflow.py` 中三个专家并行调用
- **差距**：结构化输出格式未强制约束、无不确定性标注、无升级条件输出字段

## 4. Safety Reviewer（安全审查角色）

### 职责

- 检查 Coordinator 生成的最终答复是否存在**危险建议**
- 检查是否遗漏了**升级人工医师或紧急就诊提醒**
- 检查在**高不确定性**场景下是否表现得过于肯定
- 检查答复是否与**已知患者信息冲突**（如已知过敏药物却被建议使用）
- 对高风险答复进行拦截或标注

### 边界

- Safety Reviewer 是**强约束层**，不是可有可无的装饰流程
- 它有权阻止不安全答复的输出
- 它不负责改写答复内容——如果审查不通过，回退给 Coordinator 重新生成
- 它不负责评价答复的质量（质量评价由 Evaluation Pipeline 负责）

### 当前实现状态

- **尚未独立实现**：当前代码中没有独立的 Safety Reviewer 角色
- Doctor Agent 部分承担了安全审查职能，但不是独立的审查环节
- `workflow.py` 中缺少安全审查步骤
- **这是当前最大的架构缺口之一**

---

## 5. Evaluation Pipeline（评价与改进流水线）

### 职责

- 接收每轮咨询的 consultation trace
- 接收人类专家的评分与标签（GOOD / BAD / NEUTRAL / ERROR）
- 形成 failure taxonomy（失败分类）
- 在受控范围内提出 candidate improvement
- 提供版本比较依据

### 边界

- **不能直接在线修改生产逻辑**：任何改动都走离线环
- **不能绕开人类审批上线高风险变更**
- 不信任 Agent 自评——所有质量判断来自人类
- 不能仅凭少量正面案例就宣称系统改善

### 当前实现状态

- `src/evolution/evaluation_service.py`：EvaluationService 已实现 pending/completed 生命周期管理
- 支持 GOOD/BAD/NEUTRAL/ERROR 标签、safety/personalized/advice_direction 扩展维度
- `api_server.py` 中有完整的评价 CRUD API
- **差距**：failure taxonomy 只有 4 个粗粒度标签（缺 11 种细分类型）、无回放/回归验证机制、无版本比较工具

## 6. 角色间协作流程（在线服务环）

```
用户提问
    │
    ▼
┌─────────────────┐
│   Coordinator    │ ← 重述问题、判断类型
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ Memory Governor  │ ← 检索患者上下文、组装 dossier
└────────┬────────┘
         │
         ▼
┌─────────────────────────────────────┐
│         路由决策（Coordinator）        │
│  根据问题类型选择需要的专家            │
└────┬──────────┬──────────┬─────────┘
     │          │          │
     ▼          ▼          ▼
┌─────────┐ ┌─────────┐ ┌─────────┐
│Pharmacist│ │Nutrition│ │ Doctor  │  ← 并行处理
└────┬────┘ └────┬────┘ └────┬────┘
     │          │          │
     └──────────┼──────────┘
                │
                ▼
┌─────────────────┐
│   Coordinator    │ ← 汇总专家意见、生成最终答复
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ Safety Reviewer  │ ← 安全检查（通过/拦截/标注）
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│   返回给用户      │
└────────┬────────┘
         │
         ▼
┌─────────────────────────────────────┐
│         后续处理（非阻塞）             │
│  ① Memory Governor: 写回记忆候选      │
│  ② Evaluation Pipeline: 记录 trace   │
│  ③ 创建待评价任务                      │
└─────────────────────────────────────┘
```

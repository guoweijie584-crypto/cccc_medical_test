# 面向智能客服的 AI Agent 血糖管理自进化系统

## 系统架构与业务逻辑说明

## 这份文档的定位

这份文档是给人看的主说明文档，不是 PPT 提词稿，也不是 AI 索引。

它的目标是把下面这些问题一次讲清楚：

- 这个项目为什么成立
- 系统整体是怎么设计的
- 各个 Agent 分别负责什么
- 记忆管理和自进化到底在系统里处于什么位置
- 当前到底做到哪一步

同时要明确一条写作边界：

- 本文会把 `现有文档设计` 和 `现有源码证据` 合并整理
- 但不会把仍在推进中的部分写成“已经完全实现”

## 一、项目背景与问题场景

这个项目围绕的核心问题不是一般聊天问答，而是：

`血糖管理是一个天然需要长期上下文、连续跟踪和多专业协作的场景。`

从现有材料看，团队之所以选择这个题，主要是因为血糖管理同时具有以下特点：

1. 需要长期连续管理  
   血糖控制不是一次性咨询，而是持续性的监测、判断、调整过程。

2. 需要多专业角色协同  
   同一个问题往往同时涉及用药、饮食、代谢控制、患者教育甚至心理状态。

3. 需要历史信息参与决策  
   当前建议是否合理，通常不能只看这一轮问题，还要结合病史、近期波动、近期调整记录等上下文。

所以这个项目的方向不是“用一个大模型回答问题”，而是：

- 用多 Agent 分担专业角色
- 用记忆管理保证长期连续性
- 用评测驱动的机制逐步优化提示词与记忆质量

## 二、系统整体架构

从现有 PPT、流程图和源码仓库交叉来看，系统可以分成 4 个层次理解。

### 1. 用户入口层

这一层负责：

- 接收患者咨询
- 把问题送入系统
- 触发后续协作流程

在架构图中出现过的入口包括：

- Web UI
- CLI
- IM Bridge
- MCP Server

但对人类理解来说，这一层最重要的不是具体入口形式，而是：

`用户问题进入系统后，不会被单一模型直接回答，而是会进入多 Agent 协作流程。`

### 2. 业务协作层

这是系统最核心的一层。

当前较稳定的业务角色包括：

- 主治医生 / Primary / Foreman
- 药剂师 / Pharmacist
- 营养师 / Nutritionist
- 代谢病医生 / Doctor
- 记忆管理 / Memory Agent

扩展角色在汇报主稿里也多次出现：

- 科普老师 / Educator
- 心理咨询师 / Counselor

### 3. 记忆治理层

这是系统最有辨识度的一条主线。

其核心思想不是“每个专家自己带记忆”，而是：

`由独立的 Memory Agent 统一负责上下文读取、关键事实写回和记忆治理。`

### 4. 评测与优化层

这是第二条核心主线。

它的目标不是简单改提示词，而是：

- 先评估当前输出和记忆质量
- 再判断问题到底来自 prompt、memory 还是协作逻辑
- 然后把优化任务路由给对应模块

## 三、核心 Agent 角色与职责

下面这张表是当前对人最有用的整理方式：既保留角色边界，也避免把运行层细节讲得太杂。

| 角色名称 | 英文标识 | 角色定位 | 核心职责与边界 |
| --- | --- | --- | --- |
| 主治医生 | Primary / Foreman | 主要决策者 | 负责接收问题、统筹各专家意见并输出最终建议；是患者侧统一出口。 |
| 药剂师 | Pharmacist | 辅助决策 | 负责药物风险、剂量、相互作用和安全提醒。 |
| 营养师 | Nutritionist | 辅助决策 | 负责饮食结构、餐食建议与饮食对血糖的影响分析。 |
| 代谢病医生 | Doctor | 辅助决策 | 负责代谢控制、并发症风险和疾病进展判断。 |
| 记忆管理 | Memory Agent | 系统治理 | 负责上下文读取、事实提取、记忆写回与治理，不直接做业务决策。 |
| 质检员 | Evaluator | 评测模块 | 对回答质量和记忆质量进行评分。 |
| 分析师 | Analyzer | 归因模块 | 判断低分根因属于 prompt、memory 还是 coordination。 |
| 提示词优化师 | Prompt Optimizer | 优化模块 | 针对具体 Agent 的提示词进行差量优化。 |
| 记忆优化师 | Memory Optimizer | 优化模块 | 对记忆内容做增删改查与结构调整。 |

补充说明：

- 科普老师和心理咨询师在文档主稿里是稳定出现的扩展角色
- 但在当前源码仓库里，主流程真正落到代码的核心业务专家仍然主要是 `Primary / Pharmacist / Nutritionist / Doctor / Memory`

## 四、协作与制衡逻辑

这套系统设计里，最重要的不是“角色多”，而是“角色之间有主次和边界”。

### 1. 明确最终责任人

主治医生 / Foreman 是唯一的患者侧出口。

这样做的好处是：

- 不会把多个专家的原始意见直接抛给患者
- 避免建议冲突导致理解混乱
- 保证系统对外始终有一个统一责任主体

### 2. 保留多专业制衡

辅助专家并不是装饰性角色，而是用来提供专业制衡。

例如：

- 药剂师负责药物相关判断
- 营养师负责饮食相关判断
- 代谢病医生负责代谢控制与并发症风险判断

这意味着系统并不是“一个 Agent 装成很多角色”，而是试图模拟多学科协作的会诊结构。

### 3. 记忆治理与业务决策分离

这也是整个系统非常关键的一点：

- 专家 Agent 负责分析和建议
- Memory Agent 负责上下文和事实治理

从业务逻辑上说，这种分离比“所有专家都直接碰底层记忆存储”更清楚，也更容易后续优化。

## 五、三层记忆管理架构

记忆管理是当前材料里最稳定、最反复出现的核心创新线。

### 1. 基本设计

系统把记忆分成三层：

1. 短期记忆  
   当前轮次的对话历史、本轮协作需要的即时上下文、近期中间结果。

2. 中期记忆  
   近期血糖趋势、近期饮食与运动记录、最近的用药调整等阶段性信息。

3. 长期记忆  
   患者画像、病史、过敏史、长期稳定事实和关键医疗事件。

### 2. 当前文档中的更细设计口径

在 `archive/round1_baseline/流程架构.pptx` 和相关架构材料里，三层记忆还出现过更具体的设计细节：

- 短期记忆 TTL：1 小时
- 中期记忆 TTL：30 小时
- 长期记忆：长期保留 / 永久存储
- 底层记忆引擎：SQLite + `sqlite-vec`

这里必须加一个证据边界说明：

- 这些 `TTL` 和 `sqlite-vec` 细节目前主要由流程图与设计材料支持
- 它们可以被视为当前较明确的设计方案
- 但不能直接等同于“已经全部稳定落地并验证完成”

### 3. Memory Agent 的职责

从文档与源码交叉看，Memory Agent 的职责可以归纳为：

- 按患者读取相关上下文
- 组织长中短期信息
- 从交互中提取关键事实
- 写回或更新患者相关记忆

源码仓库里也确实已经存在对应实现原型，包括：

- `retrieve_patient_context`
- `build_agent_context`
- `extract_facts_from_interaction`
- `extract_and_store`
- `update_patient_profile`

所以更稳的表述是：

`三层记忆不再只是概念页，而是已经有明确的系统角色定位，并且在源码中存在对应原型实现。`

## 六、标准业务流程

一轮完整咨询，当前最稳的业务链路可以整理成 7 步。

### Step 1. 接收消息

主治医生 / Primary 接收用户问题，并初步判断：

- 这是不是紧急风险场景
- 这次需要调用哪些专家参与

### Step 2. 读取记忆

Primary 向 Memory Agent 请求当前患者的上下文，包括：

- 短期上下文
- 中期趋势信息
- 长期画像与病史

### Step 3. 并行咨询专家

Primary 将上下文分发给专家角色，由他们并行开展分析。

当前源码主流程中已经直接体现了这一步：

- Pharmacist
- Nutritionist
- Doctor

### Step 4. 综合意见

Primary 汇总各专家的输出，做统一整合。

### Step 5. 输出建议

系统向患者输出统一建议，而不是把内部协作过程直接暴露给用户。

### Step 6. 写回事实

Memory Agent 从当前轮对话中提取关键事实，并写回记忆系统。

### Step 7. 满足条件时触发评测

当达到周期条件或评分过低时，系统在后台触发评测与优化流程。

这 7 步最重要的意义是：

- 解释为什么这是一个多 Agent 系统
- 解释为什么记忆必须单独治理
- 解释为什么后续能自然接上自进化流程

## 七、评测驱动的双闭环自进化

这一部分是项目最有野心、但也最需要保持克制表达的部分。

### 1. 质检与评分

Evaluator 的职责是两类评分：

- 回答质量评分
- 记忆质量评分

从当前代码和提示词看，回答质量关注的维度包括：

- 医学准确性
- 安全性
- 完整性
- 个性化程度
- 一致性 / 表达质量

记忆质量关注的维度包括：

- 完整性
- 准确性
- 时效性
- 相关性
- 结构性

### 2. 根因分析

Analyzer 的核心作用是：

- 不只判断“分数低”
- 而是判断“为什么低”

当前代码中已经明确区分的根因类型包括：

- `prompt`
- `memory`
- `workflow / coordination`

### 3. 路由到对应优化器

当前系统设计的优化分流逻辑是：

- 如果是 Prompt 问题，交给 Prompt Optimizer
- 如果是 Memory 问题，交给 Memory Optimizer
- 如果是 Coordination 问题，当前也更偏向回收到 Prompt 侧做修正

### 4. 重新评测

优化完成后，系统会重新跑失败用例或重新评分，判断是否达到阈值。

### 5. 触发条件

在现有流程图与配置文件中，已经出现了比较明确的触发规则：

- 每 5 轮对话触发一次评测
- 或者当评分低于 35/50 时触发

这点比纯概念更往前，因为：

- `archive/round1_baseline/流程架构.pptx` 里有对应图示
- `config/evaluation_trigger.json` 和 `config/evaluation_group_routing.json` 里也有对应配置

但仍然要明确：

- 触发规则的存在，不等于整个双闭环已经稳定跑通
- 当前更准确的状态是：`闭环机制已经进入配置与原型阶段`

## 八、当前阶段的真实状态

基于文档和源码交叉，现在更准确的判断已经不是“只有想法”，也不是“成熟系统上线”，而是：

`系统已经从纯方案设计推进到原型联调阶段。`

### 当前可以较稳确认的内容

- 多 Agent 架构与角色分工已经明确
- 三层记忆架构已经明确，且 Memory Agent 有源码原型
- 主业务工作流已经有源码原型
- Evaluator / Analyzer / Prompt Optimizer / Memory Optimizer / Self-Evolution Loop 已有代码框架
- 本地 API 和集成测试工程已经存在

### 当前仍然必须保守表达的内容

- 不能说整个系统已经稳定运行
- 不能说双闭环已经完成验证
- 不能说记忆机制已经全部成熟定型
- 不能说已有完整性能结果或医疗效果结果

### 当前已知的实现边界

从源码检查结果看，仍然存在这些边界：

- 当前仓库更像本地集成测试 / 原型联调工程
- 患者结构化数据目录在当前 checkout 中缺失，导致部分 demo 对齐测试失败
- 阶段一记忆测试与当前实现之间仍有数据格式不一致的问题
- 业务主流程验证通过的部分，当前更多还是 mock 模式与结构验证

## 九、下阶段最合理的推进方向

如果要把“下一步做什么”讲得既真实又有方向，最合理的是这几条：

1. 做实记忆管理  
   进一步明确三层记忆的写入、整合、更新与治理规则。

2. 完善评估体系  
   让 Evaluator 和 Analyzer 的评测与归因逻辑更稳定、更可执行。

3. 验证闭环链路  
   用血糖管理测试问答集去验证从“发现问题”到“优化再评测”的完整流程。

4. 把原型推进到更强的可验证状态  
   包括补齐数据、补齐流程接入、补齐结果验证。

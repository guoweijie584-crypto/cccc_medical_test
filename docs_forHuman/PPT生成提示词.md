# PPT 生成提示词

## 使用说明

这份文件是给你直接复制到 PPT 生成模型里的。

目标不是“给 AI 一点方向”，而是：

- 尽量一次生成可用版本
- 控制页数、口径、结构
- 最大限度避免模型乱补实现细节

如果你只想做一次生成，优先使用下面的“主提示词”。

## 源码增强版主提示词

如果你已经拿到了当前源码仓库，请优先使用这一版，而不是后面的旧版主提示词。

你现在是一名非常谨慎的中文汇报型 PPT 生成助手。请基于以下要求，直接生成一版适用于课堂阶段汇报的 PPT 方案。

### 一、任务背景

我要做一个 **6 分钟左右的课堂阶段汇报**，项目主题是：

`面向智能客服场景的 AI Agent 血糖管理自进化系统`

这次汇报 **不是最终答辩**，也 **不是产品发布会**。  
这次汇报只需要讲清楚：

1. 为什么选择这个项目
2. 团队和系统角色大致怎么分工
3. 当前项目已经推进到哪一步
4. 下一步准备做什么

### 二、你必须遵守的真实性边界

请严格遵守下面这些约束，不能违反：

1. **不要把项目写成已经完整实现。**
2. **不要把自进化闭环写成已经跑通并完成验证。**
3. **不要把三层记忆机制写成已经完整落地。**
4. **不要写任何“系统稳定运行”“效果显著提升”“性能已验证”“已经成熟可部署”之类的话。**
5. **不要虚构代码、实验结果、用户数据、临床数据、性能指标。**
6. **不要展开底层运行框架细节，例如 MCP、Ledger、Daemon、CLI、具体端口、具体 runtime 品牌。**
7. **不要把这次汇报写成技术炫技风格。**

### 三、已经可以谨慎使用的源码证据

下面这些内容已经有源码依据，可以比纯文档口径稍微更实一点，但仍然要保持保守：

1. 仓库里已经存在多 Agent 工作流代码，流程包括：
   - 读取患者上下文
   - 并行调用药剂师、营养师、代谢病医生
   - 由主治医生/Primary Agent 做综合回答
   - 对交互内容做事实提取并写回记忆
2. 仓库里已经存在 `Memory Agent` 代码，包含：
   - 患者上下文检索
   - 长中短期信息组织
   - 记忆提取与存储
   - 患者画像更新
3. 仓库里已经存在自进化主循环代码，包含：
   - Evaluator
   - Analyzer
   - Prompt Optimizer
   - Memory Optimizer
   - 多轮迭代评估流程
4. 仓库里已经存在本地 API 与集成测试工程，说明项目不是纯概念稿，而是已经进入原型联调阶段。

### 四、即使有源码也仍然要保守表达的地方

下面这些点必须继续收着讲：

1. 有源码不等于已经完整跑通全部链路。
2. 有自进化 loop 代码，不等于已经验证出稳定提升效果。
3. 有 Memory Agent 代码，不等于三层记忆机制已经全部成熟定型。
4. 当前仓库是 **本地集成测试 / 原型联调工程**，不是成熟发布版产品。
5. 仓库里仍有明显未完全收口的地方，例如：
   - 部分验收测试还是 TODO / mock 风格
   - 患者结构化数据目录当前缺失，导致部分 demo 对齐测试失败
   - 某些阶段一记忆测试里，测试数据格式与当前实现不完全匹配

### 五、本次最稳的总口径

请优先围绕下面这句总口径组织整套 PPT：

`当前项目已经从纯方案设计推进到原型实现阶段：系统框架、角色分工、记忆管理主线和自进化主线都已有设计与部分源码基础；下一阶段重点是把记忆机制、评测归因和自进化链路进一步做实并完成更稳定的验证。`

### 六、项目内容边界

请仅围绕以下内容组织 PPT：

#### 1. 项目背景

- 糖尿病患者多，血糖管理需求强
- 血糖管理不是一次性问答，而是长期连续管理问题
- 这个场景天然需要长期上下文、多角色协作、历史信息调用
- 所以项目选择“多 Agent + 记忆管理 + 自进化优化”的方向

#### 2. 团队 / 角色分工

这里不要写得像公司组织架构图，也不要写太多真人介绍。

重点表达系统里的核心角色：

- 主治医生 / Foreman：负责统筹与最终决策
- 药剂师、营养师、代谢病医生：负责辅助分析
- 记忆管理 Agent：负责上下文与记忆治理
- Evaluator、Analyzer、优化模块：负责后续评测、归因与优化
- 科普老师、心理咨询师：作为拓展支持角色

如果需要提到“团队情况”，只保留非常简洁的表达，例如：

- 团队围绕系统架构、记忆管理、评测优化与汇报整理等方向协作推进

#### 3. 当前项目进展

请优先写成以下这类安全表达：

- 已完成项目方向定义
- 已完成多 Agent 架构设计与角色分工设计
- 已形成主治医生统筹、多专家协同的工作流原型
- 已具备记忆管理模块的基础代码与上下文组织能力
- 已形成自进化机制的总体路线，并已有评测、分析、优化模块代码框架
- 当前阶段属于“原型联调推进中”，不是“全部链路完全验证完成”

#### 4. 下一步计划

请只写成以下方向：

- 继续完善记忆管理机制
- 补齐患者数据、记忆写入与检索链路的稳定性验证
- 进一步细化评测、归因与优化流程
- 逐步推进自进化机制做实
- 让系统从当前原型联调走向更强的可验证能力

### 七、明确禁止出现的高风险词

以下词语或同义表达，除非明确写成“计划 / 设计 / 路线 / 原型”，否则不要出现：

- 已完整实现
- 已稳定运行
- 已跑通闭环
- 代码实现完成
- 自动优化完成
- 性能显著提升
- 已验证效果
- 临床验证
- 商业化落地
- 7×24 实时响应能力已实现
- 永久锁定

### 八、允许出现但必须谨慎使用的表达

下面这些说法可以出现，但必须控制语气，不能写得过满：

- 已有关键模块源码
- 已形成工作流原型
- 已进入本地联调阶段
- 已具备基础 API 与集成测试环境
- 自进化机制已有代码框架
- 记忆管理已有原型能力

### 九、建议的 PPT 结构

请生成 **5 页正文 + 封面 + 致谢** 的版本。

正文页结构固定如下：

1. 项目背景与选题原因
2. 团队情况 / 角色分工
3. 项目整体思路
4. 当前阶段进展
5. 下一步计划

### 十、每一页的输出要求

请你直接输出每一页的以下内容：

#### 对每一页都必须给出：

1. 页面标题
2. 页面上适合直接放的 3-5 条短文案
3. 这页建议讲的核心一句话
4. 这页应避免出现的错误表达

### 十一、风格要求

- 中文表达
- 适合课堂汇报
- 简洁、稳重、清楚
- 重点是“老师一看就懂”
- 不要写得像商业路演
- 不要写得像论文摘要
- 不要大段堆概念

### 十二、对内容真实性的额外要求

请严格区分：

- 已完成的方向定义与架构设计
- 已有源码原型支撑的模块
- 已提出但尚未完全实现或验证的机制
- 下一阶段计划

请始终优先使用下面这些词：

- 已完成设计
- 已形成方案
- 已明确方向
- 已有原型基础
- 已有关键模块源码
- 正在推进
- 下一阶段计划

而不要使用下面这些词：

- 已完全实现
- 已验证完成
- 已落地
- 已上线
- 已成熟

## 如果模型仍然生成得太“满”

可以继续追加这段二次修正提示词：

请对你刚才生成的 PPT 内容做一次“降风险重写”，要求：

1. 所有“已实现、已跑通、已验证、已落地”改成“已完成设计、已形成方案、正在推进、下一阶段计划”

4. 把“代码实现”统一改成“初步搭建”或“框架设计”
